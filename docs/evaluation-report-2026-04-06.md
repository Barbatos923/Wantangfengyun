# 《晚唐风云》项目质量评估与架构优化报告 (2026-04-06)

## 0. 概述
本报告基于外部代码审查（Kimi）及内部深度审计，对《晚唐风云》项目 UI 层与引擎层的全量代码进行了质量评估。当前项目处于快速功能迭代后的“技术债积累期”，存在若干影响长期稳定性与性能的严重架构问题。

---

## 1. 核心风险评估 (已验证)

### 1.1 隐式循环依赖与 Tree-shaking 风险 ⚠️ **高危 (P0)**
*   **位置**: `engine/npc/NpcEngine.ts` (L20-L50)
*   **现状**: 通过 30+ 行副作用导入（`import './behaviors/xxx'`）触发行为注册。
*   **影响**: 生产环境打包时，若无导出引用，Tree-shaking 可能移除这些行为，导致 NPC 逻辑静默失效；IDE 无法静态分析行为调用链。
*   **建议**: 改为显式注册模式，在引擎初始化阶段调用 `registerBehaviors([...])`。

### 1.2 Store 状态更新性能瓶颈 ⚠️ **中高危 (P1)**
*   **位置**: `CharacterStore.ts`, `TerritoryStore.ts`
*   **现状**: 每次 `updateCharacter` 均执行 `new Map(state.characters)`（全量拷贝）。
*   **影响**: 在 1000+ 角色规模下，月结批量更新将触发密集的 GC 压力，导致掉帧（Jank）。
*   **建议**: 引入 `immer` 或类似的结构共享优化机制，将单次全量拷贝优化为局部更新。

### 1.3 随机数确定性缺失 ⚠️ **中危 (P1)**
*   **位置**: `TurnManager.ts` (L60), `WarStore.ts`
*   **现状**: 使用 `Date.now()` 作为种子且未持久化；多处直接使用 `crypto.randomUUID()`。
*   **影响**: 存档/读档后游戏走向无法重现，破坏策略游戏的公平性与调试一致性。
*   **建议**: 将 RNG 状态存入 Store 并确保存档包含种子；使用基于种子的 ID 生成算法。

### 1.4 回调注册生命周期泄漏 ⚠️ **中危 (P2)**
*   **位置**: `TurnManager.ts` (L42-44)
*   **现状**: 回调存储在模块级变量 `Map` 中，而非 Store 内部。
*   **影响**: HMR（热更新）时旧回调不清理；组件卸载若未注销将导致内存泄漏；存档无法恢复函数状态。
*   **建议**: 将回调注册表整合入系统管理器，并与 React 组件生命周期严格绑定。

## 2. 架构设计缺陷 (已验证)

### 2.1 魔法数字与业务解耦不足
*   **位置**: `isActiveMonth` (NpcEngine.ts)
*   **问题**: 硬编码 `rankLevel >= 17`（节度使）和 `12`（刺史）。
*   **风险**: 若 `ranks.ts` 中的职级权重调整，逻辑将静默失效。
*   **建议**: 提取为命名常量或从配置表中读取阈值。

### 2.2 数据索引缺失导致 O(n) 查询
*   **问题**: `WarStore` 和 `MilitaryStore` 频繁遍历 `Map.values()`。
*   **影响**: 随着战争数量增加，查询性能线性下降。
*   **建议**: 建立双向索引（如 `charId -> warIds`），提升查询效率至 O(1)。

## 3. 测试与代码质量

### 3.1 测试覆盖率严重不足 ⚠️ **高危 (P0)**
*   **现状**: 核心系统（NPC 决策、战争围城、继承逻辑）单元测试缺失率 > 90%。
*   **现状**: UI 组件完全无自动化测试（Vitest/Testing Library）。

### 3.2 生产环境调试代码残留
*   **现状**: 存在 35+ 处 `console.log/error`，部分输出包含敏感逻辑。
*   **建议**: 封装统一的 `Logger` 类，根据 `ENV` 自动屏蔽输出。
