# 晚唐风云：继承法与人事任命系统设计方案 - Manus

> 本文档为交付 Claude 实施的完整设计规格，涵盖数据结构扩展、核心机制逻辑、边界条件处理及实施范围划定。

---

## 一、 核心设计理念

本系统旨在模拟晚唐时期**中央集权的流官体系**与**藩镇割据的宗法世袭**之间的政治博弈。设计的核心原则是将继承法与人事任命权**绑定在具体岗位（`Post`）上**，而非领地或角色上。通过引入"辟署权"这一关键概念，清晰地划分出朝廷管辖区与藩镇私有领地的人事边界。

## 二、 数据结构扩展

### 2.1 `Post` 接口扩展

在 `src/engine/territory/types.ts` 的 `Post` 接口上新增以下字段：

```typescript
export interface Post {
  // ===== 现有字段（保持不变）=====
  id: string;
  templateId: string;
  territoryId?: string;
  holderId: string | null;
  appointedBy?: string;
  appointedDate?: GameDate;

  // ===== 新增字段 =====
  successionLaw: 'clan' | 'bureaucratic';
  // 继承法类型：'clan' = 宗法继承，'bureaucratic' = 流官继承

  hasAppointRight: boolean;
  // 辟署权：true 表示该岗位持有人对其辖区内所有岗位拥有绝对任命权

  designatedHeirId?: string | null;
  // 留后：预先指定的宗法继承人 ID（仅 successionLaw === 'clan' 时有效）

  isActing?: boolean;
  // 权知标记：true 表示当前持有人为降格代理任命，非正式在任

  minRankOverride?: RankLevel;
  // 品级覆盖：若存在，覆盖模板（PositionTemplate）中的 minRank。
  // 用于区分上州/中州/下州刺史等同模板但品级不同的岗位实例。
  // 所有读取 minRank 的代码均应改为：post.minRankOverride ?? template.minRank
}
```

### 2.2 初始化默认值规则

在 `src/data/territories.ts` 初始化各领地的岗位实例时，按以下规则设置新字段的默认值：

| 岗位类型 | `successionLaw` | `hasAppointRight` |
|---|---|---|
| 皇帝（`pos-emperor`） | `'clan'` | `true` |
| 诸侯王（`pos-wang`）、行台尚书令 | `'clan'` | `true` |
| 节度使（`pos-jiedushi`）、观察使（`pos-guanchashi`） | `'clan'` | `false`（初始，可通过政体面板修改） |
| 防御使（`pos-fangyu-shi`）、刺史（`pos-cishi`） | `'bureaucratic'` | `false` |
| 所有副岗位（`grantsControl: false`） | `'bureaucratic'` | `false` |

## 三、 辟署权机制

### 3.1 权力定义

拥有辟署权（`hasAppointRight: true`）的岗位持有人，在其领地边界内（含所有子领地），对**全部岗位**拥有绝对的任命权与罢免权。这是一道**硬性防火墙**：一旦某领地处于辟署权保护之下，外部的吏部、宰相乃至皇帝，均无权对该领地内的任何岗位进行任命或罢免操作（完全隔离，无例外）。

### 3.2 辟署权的判断逻辑

在 `canAppointToPost()` 和 `getAppointableVacantPosts()` 中，任命前须执行以下检查：

1. 取待任命岗位的 `territoryId`，找到对应领地。
2. 检查该领地本身，以及其 `parentId` 所指向的父级领地，其**主岗位（`grantsControl: true`）的持有人**是否拥有辟署权（`hasAppointRight: true`）。
3. 若是，则**仅该持有人**有权任命此岗位，其他任何人的任命请求均返回失败。
4. 若否，则进入流官任命体系（见第五节）。

### 3.3 辟署权的嵌套授权

辟署权可以向下嵌套授权，形成层级结构：

```
皇帝（天下辟署权，先天拥有）
  └─ 诸侯王（国级辟署权，皇帝授予）
       └─ 节度使（道级辟署权，诸侯王或皇帝授予）
```

每一级辟署权持有人只需管理其**直接下属主岗位**的任命，不需要越级管理。

### 3.4 辟署权的变更

辟署权可通过政体面板修改（`hasAppointRight` 字段直接更新）。取消辟署权是严重的政治操作，应触发对应角色的好感度大幅减损，后续可扩展为叛乱触发条件。

## 四、 宗法继承机制

适用于 `successionLaw === 'clan'` 的岗位。

### 4.1 触发时机

岗位持有人死亡（`alive` 变为 `false`）时，在 `characterSystem.ts` 的死亡处理逻辑中**立即触发**，不等到下一回合。

### 4.2 继承顺位

按以下优先级依次检查：

1. **留后**：若 `designatedHeirId` 不为空，且该角色存活，则直接继承，无需其他判断。
2. **嫡长子**：`family.childrenIds` 中，`family.isLegitimate === true` 且年龄最长的存活子嗣，继承时附加**正统性加成**（具体数值待后续正统性系统设计）。
3. **其他宗族成员**：同 `clan` 字段相同、存活、且 `overlordId` 追溯链指向当前持有人的角色，按血缘远近排序。血缘疏远者附加**正统性减益**。

### 4.3 绝嗣处理

若以上三步均无合法继承人，该岗位**上交**给其法理上的上级：

- 节度使岗位上交给其父级领地的主岗位持有人（诸侯王或皇帝）。
- 诸侯王岗位上交给皇帝。
- 皇帝岗位上交逻辑待单独设计。

上交后，该岗位的 `successionLaw` 由接收方决定是否保留或改为 `'bureaucratic'`。

## 五、 流官继承机制

适用于 `successionLaw === 'bureaucratic'` 的岗位。

### 5.1 触发场景与处理节奏

| 触发场景 | 处理方式 | 空窗期影响 |
|---|---|---|
| **岗位持有人死亡**（`grantsControl: true`） | 死亡当月产生"待处理铨选事件"，推入对应任命者的事件队列，下一回合由玩家/AI确认。 | 地方主岗位领地 `control` 和产出开始下降（因 `getActualController` 返回 `null`，`rulerAbilities` 退化为默认值）。 副岗位及京官目前无实质数值影响（待第六阶段完善）|
| **因平调、升职、免官和未及时任命导致的岗位空缺** | 推入对应任命者的事件队列，下一回合由玩家/AI确认。 | 同上。 |

### 5.2 分级任命权

流官体系中，任命权按岗位品级分级归属。系统触发铨选时，自动查询当前实际持有相应岗位的角色，将铨选单推送给对应的任命者。

**注意**：品级比较时，应优先使用 `post.minRankOverride ?? template.minRank`，而非直接使用 `template.minRank`。

| 品级范围 | 任命权归属 | 查询岗位 ID |
|---|---|---|
| **五品以下**（有效 `minRank < 18`） | 吏部尚书 | `pos-guanlibu-shangshu` |
| **五品以上**（有效 `minRank ≥ 18`） | 宰相 | `pos-zaixiang` |
| **特旨**（任意品级） | 皇帝 | `pos-emperor` |

皇帝特旨权可无视品级强行任免，但频繁行使应触发宰相好感度减损（具体数值待后续设计）。

**辟署权范围内的镜像体系**：拥有辟署权的诸侯王或节度使，在其辖区内适用相同的分级逻辑。其辖区内高级岗位（有效 `minRank ≥ 18`）由其本人负责任命；低级岗位（有效 `minRank < 18`）由其幕府的**节度判官**（`pos-jiedushi-panguan`）执行自动铨选——系统应查询该节度使辖区内 `pos-jiedushi-panguan` 的持有人，而非 `pos-guanlibu-shangshu`。若节度判官岗位空缺，则由节度使本人兜底处理。

### 5.3 候选人池筛选逻辑

触发铨选时，系统自动生成候选人池。所有候选人须满足以下**共同硬性前提**：

- `alive: true` 且 `official` 字段存在。
- `overlordId` 追溯链最终指向当前任命者（效忠于任命者）。
- 角色自身 `rankLevel` ≥ 空缺岗位的有效 `minRank`。

满足前提后，候选人按以下**三个层次**分组，展示时分组显示，系统默认优先推荐第一层：

| 层次 | 名称 | 筛选条件 |
|---|---|---|
| **第一层** | 升调 | 候选人当前岗位的有效 `minRank` **严格低于**空缺岗位的有效 `minRank` |
| **第二层** | 平调 | 候选人当前岗位的有效 `minRank` **等于**空缺岗位的有效 `minRank`，且不是同一个岗位 |
| **第三层** | 新授 | 候选人当前**无岗位** |

对于**平调**候选人和**升调**候选人，选择后需要连带处理原岗位空缺。可选择暂时令岗位空缺。

### 5.4 候选人综合评分与推荐

系统对候选池中所有角色进行综合评分，**最高分者作为系统默认推荐人选**（AI 直接采用此人选）：

```
综合分 = 贤能值（virtue）× 0.4
       + 岗位匹配能力值 × 0.2   （文职取 administration，武职取 military）
       + 考课加值 × 0.4          （无考课记录的角色此项为 0）
```

*未来扩展：可在公式中加入好感度权重（党争因子），无需改变框架，只需调整系数。*

### 5.5 铨选单交互（UI 设计）

- 玩家（皇帝、宰相、吏部尚书，或拥有辟署权的诸侯王/节度使）收到一份**铨选单**。
- 铨选单列出所有空缺岗位，每行已填入系统推荐的候选人，玩家可**一键全部确认**。
- 若玩家不满意某一推荐，可点击该行展开完整候选人池（分三层显示），手动选择其他人选。

### 5.6 权知机制（无人可继承时的降格处理）

若候选人池为空（三层均无人满足品位要求）：

- **权知代理**：系统寻找 `rankLevel` 略低于有效 `minRank` 的角色，进行降格代理任命，设置 `isActing: true`。权知官员受到轻微的正统性或贤能积累惩罚（具体数值待后续设计）。
- **长期空缺**：若连权知人选都没有，岗位保持 `holderId: null`。每月结算时系统自动重新检查候选池，直到有合适人选出现。

## 六、 三年一考机制（考课系统）

### 6.1 触发方式

采用**统一考课**模式：每三年正月（`date.month === 1` 且 `date.year % 3 === 0`）统一触发。

### 6.2 适用范围

- 朝廷直辖官员：由吏部/宰相执行考课。
- 辟署权范围内的官员：由辟署权持有人（节度使/诸侯王）自行执行考课，朝廷不介入。

### 6.3 考课评分

综合以下维度计算考课分数（0–100）：

- 领地治理成效：三年内该官员控制领地的 `control`、`development` 净增长量（权重高）。
- 贤能积累：`official.virtue` 的三年增长量（权重中）。
- 岗位匹配度：文职取 `administration`，武职取 `military`（权重中）。

### 6.4 考课结果处理

| 分数段 | 结果 | 后续操作 |
|---|---|---|
| ≥ 80 | 上等，升迁候选 | 铨选单中有较高加值 |
| 60–79 | 中等，原职留任 | 无需操作，自动续任，铨选单中无加值 |
| < 60 | 下等，罢免 | 触发补缺流程，原岗位空缺，进入铨选单 |

考课产生的罢免空缺，与当月常规空缺**合并**为一张铨选单，由玩家/AI 一次性审批。

## 七、 失职官员的数据流向规则

### 7.1 核心原则

`overlordId` 存储的始终是**角色 ID（Character ID）**，记录的是"任命他的人"，而非他所管辖的领地。代码实现时，需通过 `getPostsByTemplate('pos-emperor')[0]?.holderId` 等方式动态查询当前持有者的角色 ID，**不得硬编码岗位 ID**。

当官员失去岗位时，其 `overlordId` 应**回归到任命者本人**，而不是飘散为无主状态。

### 7.2 分场景处理规则

| 场景 | 罢免者 | 罢免后 `overlordId` 归属 | 说明 |
|---|---|---|---|
| 朝廷流官被吏部/宰相罢免 | 吏部尚书或宰相 | 指向**当前皇帝的角色 ID** | 回归中央人才池，成为"待选"状态。 |
| 辟署权领地内的官员被节度使罢免 | 节度使 | 指向**节度使本人的角色 ID** | 成为节度使的"在野幕僚"，留在藩镇人才池，不流向中央。 |
| 辟署权领地内的官员被诸侯王罢免 | 诸侯王 | 指向**诸侯王本人的角色 ID** | 同上，留在诸侯国人才池。 |
| 宗法继承失败（无继承人，岗位上交） | 系统 | 指向**上级法理所属的角色 ID** | 绝嗣藩镇的旧幕僚随岗位一并归入上级人才池。 |

### 7.3 死亡角色的处理

角色死亡（`alive: false`）后，`overlordId` 无需清空，保留原值即可。候选人池筛选时首先过滤 `alive: true`，死亡角色自然不进入候选池。

### 7.4 防止"流浪汉"堆积的兜底机制

月度结算时，对无岗位且存活的角色执行以下清理：

- **年龄 ≥ 70 岁且无岗位**：触发自然退休，角色标记为 `retired: true`，从所有候选池中永久移除，数据保留用于历史记录。
- **无岗位超过 10 年**：触发一次健康检定，加速自然死亡概率，模拟贫困流亡导致的死亡。

## 九、 第一版实施范围

### 纳入范围（本次实施）

- `Post` 接口扩展（`successionLaw`、`hasAppointRight`、`designatedHeirId`、`isActing`、`minRankOverride`）。
- `territories.ts` 初始化数据中为各岗位实例设置新字段默认值。
- 辟署权对 `canAppointToPost()` 和 `getAppointableVacantPosts()` 的硬性拦截。
- 主岗位（`grantsControl: true`）的**宗法继承**逻辑（含留后、顺位继承、绝嗣上交）。
- 主岗位的**流官继承**逻辑（含候选人池三层筛选、综合评分、权知机制）。
- 分级任命权（吏部/宰相/皇帝/节度判官）的查询与铨选单推送。
- 失职官员 `overlordId` 的正确更新（修复现有 Bug）。
- 铨选单 UI 雏形（推荐候选人 + 可展开手动覆盖）。

### 暂不处理（第六阶段）

- 副岗位（`grantsControl: false`）的继承逻辑。
- 京官的党争逻辑与 AI 宰相/吏部的派系行为。
- 三年一考的完整实现。
- 正统性系统（宗法继承的加成/减益数值）。
- 辟署权变更的政治后果（叛乱触发条件）。
