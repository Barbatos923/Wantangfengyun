## 已确认问题

### 1. 战争领袖死亡后，战争不会把领袖位移交给继承人或上交接收人，后续可能被错误白和平

- 现象：
  - 角色死亡时，会正常处理岗位继承、绝嗣上交、附庸转移与军队继承。
  - 但如果死者正是战争的 `attackerId` 或 `defenderId`，战争侧不会同步切换领袖。
- 原因：
  - [characterSystem.ts](/D:/桌面/Github上传/Wantangfengyun/wantang/src/engine/systems/characterSystem.ts#L312) 只会在死亡后：
    - 解散其行营
    - `removeParticipant(deadId)`
  - [WarStore.ts](/D:/桌面/Github上传/Wantangfengyun/wantang/src/engine/military/WarStore.ts#L149) 的 `removeParticipant()` 只会从 `attackerParticipants / defenderParticipants` 中删除，不会动 `war.attackerId / war.defenderId`。
  - 于是 [warSystem.ts](/D:/桌面/Github上传/Wantangfengyun/wantang/src/engine/systems/warSystem.ts#L631) 之后的自动结算，仍会按旧领袖去算 `getRealmZhouCount()`。
  - 而死亡继承已经把头衔从死者身上转走，死者一侧很容易被算成 `0` 州，接着在 [warSystem.ts](/D:/桌面/Github上传/Wantangfengyun/wantang/src/engine/systems/warSystem.ts#L634) 被直接白和平。
- 影响：
  - 本来应该由继承人继续的战争，会在领袖死亡后被系统静默洗成白和平。
  - 本来应该由“绝嗣上交”接收人继续的战争，也会被提前结束。
- 涉及文件：
  - [characterSystem.ts](/D:/桌面/Github上传/Wantangfengyun/wantang/src/engine/systems/characterSystem.ts#L312)
  - [WarStore.ts](/D:/桌面/Github上传/Wantangfengyun/wantang/src/engine/military/WarStore.ts#L149)
  - [warParticipantUtils.ts](/D:/桌面/Github上传/Wantangfengyun/wantang/src/engine/military/warParticipantUtils.ts#L6)
  - [warSystem.ts](/D:/桌面/Github上传/Wantangfengyun/wantang/src/engine/systems/warSystem.ts#L631)
  - [postQueries.ts](/D:/桌面/Github上传/Wantangfengyun/wantang/src/engine/official/postQueries.ts#L179)

### 2. 玩家角色若绝嗣死亡，`playerId` 与 `isPlayer` 不会被清掉，系统会继续把死人当玩家

- 现象：
  - 玩家角色死亡且有继承人时，系统会正确切换玩家位。
  - 但玩家绝嗣死亡时，只会发出“王朝覆灭”事件，不会清理玩家标记。
- 原因：
  - [characterSystem.ts](/D:/桌面/Github上传/Wantangfengyun/wantang/src/engine/systems/characterSystem.ts#L290) 只有 `primaryHeir` 存在时才会：
    - `setPlayerId(primaryHeir)`
    - 把继承人设为 `isPlayer: true`
    - 把死者设为 `isPlayer: false`
  - 无继承人分支只发事件，不清 `playerId`，也不清死者的 `isPlayer`。
  - [CharacterStore.ts](/D:/桌面/Github上传/Wantangfengyun/wantang/src/engine/character/CharacterStore.ts#L81) 的 `getPlayer()` 只是按 `playerId` 取角色，不检查是否存活。
  - 很多 UI 和交互入口也只看 `playerId` 是否存在，比如：
    - [BottomBar.tsx](/D:/桌面/Github上传/Wantangfengyun/wantang/src/ui/components/BottomBar.tsx#L13)
    - [DecisionPanel.tsx](/D:/桌面/Github上传/Wantangfengyun/wantang/src/ui/components/DecisionPanel.tsx#L19)
    - [CampaignPopup.tsx](/D:/桌面/Github上传/Wantangfengyun/wantang/src/ui/components/CampaignPopup.tsx#L31)
- 影响：
  - 会进入一种“王朝覆灭了，但玩家位还挂在死人身上”的半死状态。
  - 后续 UI、NPC、交互入口都可能继续围绕一个死人角色运转。
- 涉及文件：
  - [characterSystem.ts](/D:/桌面/Github上传/Wantangfengyun/wantang/src/engine/systems/characterSystem.ts#L290)
  - [CharacterStore.ts](/D:/桌面/Github上传/Wantangfengyun/wantang/src/engine/character/CharacterStore.ts#L81)
  - [BottomBar.tsx](/D:/桌面/Github上传/Wantangfengyun/wantang/src/ui/components/BottomBar.tsx#L13)
  - [DecisionPanel.tsx](/D:/桌面/Github上传/Wantangfengyun/wantang/src/ui/components/DecisionPanel.tsx#L19)
  - [CampaignPopup.tsx](/D:/桌面/Github上传/Wantangfengyun/wantang/src/ui/components/CampaignPopup.tsx#L31)

### 3. `CharacterPanel` 的“要求效忠”预览弹窗确认时没有重新校验合法性，旧弹窗可能把已失效关系硬写回去

- 现象：
  - 玩家先在人物面板里打开“要求效忠”概率预览弹窗。
  - 弹窗打开后，如果局势发生变化，确认按钮仍会直接执行。
- 原因：
  - [CharacterPanel.tsx](/D:/桌面/Github上传/Wantangfengyun/wantang/src/ui/components/CharacterPanel.tsx#L617) 打开预览时一次性计算概率。
  - [CharacterPanel.tsx](/D:/桌面/Github上传/Wantangfengyun/wantang/src/ui/components/CharacterPanel.tsx#L924) 确认时直接调用 `executeDemandFealty(...)`。
  - 但 [demandFealtyAction.ts](/D:/桌面/Github上传/Wantangfengyun/wantang/src/engine/interaction/demandFealtyAction.ts#L255) 没有重跑 `canDemandFealty / canDemandFealtyPure`，成功后会直接 `updateCharacter(targetId, { overlordId: playerId })`。
- 影响：
  - 只要弹窗打开后关系已失效，旧弹窗仍可能把目标强行挂回玩家的效忠链。

### 4. `CharacterPanel` 的“归附”预览弹窗确认时没有重新校验“仍独立 / 层级仍成立 / 仍相邻”等条件

- 现象：
  - 玩家先在人物面板打开“归附”概率预览。
  - 弹窗停留期间若世界状态变化，确认后仍可能落地旧结果。
- 原因：
  - [CharacterPanel.tsx](/D:/桌面/Github上传/Wantangfengyun/wantang/src/ui/components/CharacterPanel.tsx#L621) 打开预览。
  - [CharacterPanel.tsx](/D:/桌面/Github上传/Wantangfengyun/wantang/src/ui/components/CharacterPanel.tsx#L965) 确认时直接调用 `executePledgeAllegiance(...)`。
  - [pledgeAllegianceAction.ts](/D:/桌面/Github上传/Wantangfengyun/wantang/src/engine/interaction/pledgeAllegianceAction.ts#L297) 会重算概率，但不会重跑 `canPledgeAllegiance / canPledgeAllegiancePure`；成功时直接把 `player.overlordId = targetId`。
- 影响：
  - 弹窗打开后，即使目标已不再独立、主权层级变化、领地不再相邻，旧弹窗仍可能把玩家塞进一条已经不合法的效忠链。

### 5. `CharacterPanel` 的“召集参战”即时弹窗仍绕过了后来补上的 `validateCallToArms(...)`

- 现象：
  - `PlayerTask` 路线已经有“过期前提重校验”。
  - 但人物面板里的即时弹窗路线还没有统一过去。
- 原因：
  - [CharacterPanel.tsx](/D:/桌面/Github上传/Wantangfengyun/wantang/src/ui/components/CharacterPanel.tsx#L849) 打开召集预览。
  - [CharacterPanel.tsx](/D:/桌面/Github上传/Wantangfengyun/wantang/src/ui/components/CharacterPanel.tsx#L883) 确认时直接调用 `executeCallToArms(...)`。
  - [joinWarAction.ts](/D:/桌面/Github上传/Wantangfengyun/wantang/src/engine/interaction/joinWarAction.ts#L129) 的 `executeCallToArms(...)` 只重算概率，成功时直接 `executeJoinWar(...)`。
  - 你们后来专门补的 [validateCallToArms(...)](/D:/桌面/Github上传/Wantangfengyun/wantang/src/engine/interaction/joinWarAction.ts#L153) 目前只用于异步任务入口。
- 影响：
  - 即时弹窗路径仍可能在前提已失效后，把旧召集结果落到战争状态里。

### 6. `CharacterPanel` 的“议定进奉”确认时没有重跑合法性校验，旧弹窗可能在关系变化后改动税级

- 现象：
  - 玩家在人物面板里选“提高/降低”方向并看到概率后，若局势变化，确认仍可能按旧前提继续。
- 原因：
  - [CharacterPanel.tsx](/D:/桌面/Github上传/Wantangfengyun/wantang/src/ui/components/CharacterPanel.tsx#L743) 和 [CharacterPanel.tsx](/D:/桌面/Github上传/Wantangfengyun/wantang/src/ui/components/CharacterPanel.tsx#L753) 打开预览。
  - [CharacterPanel.tsx](/D:/桌面/Github上传/Wantangfengyun/wantang/src/ui/components/CharacterPanel.tsx#L721) 确认时直接调用 `executeNegotiateTax(...)`。
  - [negotiateTaxAction.ts](/D:/桌面/Github上传/Wantangfengyun/wantang/src/engine/interaction/negotiateTaxAction.ts#L192) 只重算概率，不重跑 `canNegotiateTax / canNegotiateTaxPure`；成功时直接 `executeTaxChange(...)`。
- 影响：
  - 旧弹窗仍可能在从属关系或资格条件变化后继续改税级，只是因为 `executeTaxChange(...)` 会 `clamp`，破坏性略低于前几条。

### 7. `DemandRightsFlow` 的预览弹窗确认时没有重新校验“仍可索权”，旧弹窗可能在资格失效后继续改权柄

- 现象：
  - 玩家先在“逼迫授权”里选定岗位与权柄，看到成功率后再确认。
  - 弹窗停留期间若从属关系、候选岗位或可索权集合发生变化，确认仍会按旧选择继续执行。
- 原因：
  - [DemandRightsFlow.tsx](/D:/桌面/Github上传/Wantangfengyun/wantang/src/ui/components/DemandRightsFlow.tsx#L37) 直接把当前 `postId/right` 交给 `executeDemandRights(...)`。
  - [demandRightsAction.ts](/D:/桌面/Github上传/Wantangfengyun/wantang/src/engine/interaction/demandRightsAction.ts#L292) 会重算概率，但不会重跑 `canDemandRights / canDemandRightsPure`，成功时直接改辟署权或继承法。
- 影响：
  - 旧弹窗可能在“已不再可逼迫授权”的局势下，继续把权柄状态写回系统。

### 8. `RevokeFlow` 里若目标岗位在弹窗停留期间已变化，执行层可能把 no-op 当成成功展示给玩家

- 现象：
  - 玩家在“剥夺领地”里选定岗位后，若该岗位在确认前已经空缺、转手或失效，执行结果仍可能显示“成功”。
- 原因：
  - [RevokeFlow.tsx](/D:/桌面/Github上传/Wantangfengyun/wantang/src/ui/components/RevokeFlow.tsx#L30) 确认时直接调用 `executeRevoke(post.id, player.id)`。
  - [revokeAction.ts](/D:/桌面/Github上传/Wantangfengyun/wantang/src/engine/interaction/revokeAction.ts#L155) 在 `!post || !post.holderId` 时直接 `return true`，没有把“岗位已失效”区分成失败或过期。
- 影响：
  - 玩家会看到“剥夺成功”，但世界状态可能根本没有发生变化，属于典型的过期弹窗误报成功。

### 9. `ReassignFlow` 预览弹窗确认时没有重新校验候选人与岗位状态，旧弹窗可能在资格变化后继续调任

- 现象：
  - 玩家先在“外放内调”里选好调任对象，看到成功率和结果预览后再确认。
  - 弹窗停留期间如果地方主岗、京官候选或皇帝/宰相关系变化，确认仍会继续走旧方案。
- 原因：
  - [ReassignFlow.tsx](/D:/桌面/Github上传/Wantangfengyun/wantang/src/ui/components/ReassignFlow.tsx#L92) 在确认时只根据当前本地选中的 `territorialPostId / replacementId` 调用 `executeReassign(...)` 或 `submitReassignProposal(...)`。
  - [reassignAction.ts](/D:/桌面/Github上传/Wantangfengyun/wantang/src/engine/interaction/reassignAction.ts#L402) 进入执行后没有统一重跑“这次调任现在是否仍合法”的完整资格检查，只会继续算成功率并落地。
- 影响：
  - 旧弹窗可能在岗位归属、候选身份或制度前提变化后，继续触发调任或叛乱分支。

### 10. `DeclareWarFlow` 的合法性主要停留在 UI 侧评估，执行层缺少对战理由仍然成立的重校验

- 现象：
  - 玩家在宣战面板里选好战由和目标后，如果弹窗打开期间停战、邻接、法理目标或资源条件发生变化，确认仍可能按旧选择继续。
- 原因：
  - [DeclareWarFlow.tsx](/D:/桌面/Github上传/Wantangfengyun/wantang/src/ui/components/DeclareWarFlow.tsx#L61) 的 `handleConfirm()` 只是检查当前面板本地状态，然后直接 `executeDeclareWar(...)`。
  - [declareWarAction.ts](/D:/桌面/Github上传/Wantangfengyun/wantang/src/engine/interaction/declareWarAction.ts#L24) 执行层只校验资源是否足够，不重跑战由、停战期、目标合法性等条件。
- 影响：
  - 这会留下“面板打开时合法，确认时局势已变，却仍然成功宣战”的窗口，属于和 `Decision` 系统旧问题同一类型的状态漂移。

### 11. `TransferVassalFlow` 没有执行前重校验，旧弹窗可能直接改写已经失效的附庸关系

- 现象：
  - 玩家在“转移臣属”弹窗里选好臣属后，若目标或该臣属的从属关系在确认前已变化，点击仍会直接生效。
- 原因：
  - [TransferVassalFlow.tsx](/D:/桌面/Github上传/Wantangfengyun/wantang/src/ui/components/TransferVassalFlow.tsx#L17) 直接调用 `executeTransferVassal(...)`。
  - [transferVassalAction.ts](/D:/桌面/Github上传/Wantangfengyun/wantang/src/engine/interaction/transferVassalAction.ts#L127) 进入执行后没有重新确认：
    - `vassalId` 仍是可转移的直属臣属
    - `newOverlordId` 仍是当前可接受转移的对象
    - `transferrerId` 仍有资格发起这次转移
  - 函数一开始就直接 `updateCharacter(vassalId, { overlordId: newOverlordId })`。
- 影响：
  - 旧弹窗可能在关系已变化后，继续把臣属链写成过期状态。

### 12. `UsurpPostFlow` 在岗位或资源状态变化后仍会显示“篡夺成功”，但执行层可能其实什么都没做

- 现象：
  - 玩家在“篡夺头衔”里选好岗位并确认后，若该岗位在弹窗停留期间已经失效、换手，或当前资源已不再足够，界面仍会进入“篡夺成功”结果页。
- 原因：
  - [UsurpPostFlow.tsx](/D:/桌面/Github上传/Wantangfengyun/wantang/src/ui/components/UsurpPostFlow.tsx#L30) 的 `handleUsurp()` 无条件 `setExecuted(true)`。
  - 但 [usurpPostAction.ts](/D:/桌面/Github上传/Wantangfengyun/wantang/src/engine/interaction/usurpPostAction.ts#L117) 只要 `!post || !post.holderId` 就直接 `return`，也没有重跑“当前是否仍可篡夺 / 资源是否仍足够”的完整资格检查。
- 影响：
  - 这会把“已经过期或已无效的篡夺操作”误报成成功，属于实质 no-op 被当成成功反馈。

### 13. `DismissFlow` 仍然主要靠打开弹窗时的候选集守门，执行层没有统一重校验当前岗位是否仍可由玩家罢免

- 现象：
  - 玩家先选定可罢免岗位，若岗位在确认前已经换手或自己失去资格，确认时仍会直接调用执行层。
- 原因：
  - [DismissFlow.tsx](/D:/桌面/Github上传/Wantangfengyun/wantang/src/ui/components/DismissFlow.tsx#L28) 直接调用 `executeDismiss(post.id, player.id)`。
  - [dismissAction.ts](/D:/桌面/Github上传/Wantangfengyun/wantang/src/engine/interaction/dismissAction.ts#L54) 没有重跑“当前玩家是否仍可罢免该岗位”的资格检查，只是基于当前 `postId` 做岗位流转。
- 影响：
  - 比起前几条，它更像“旧弹窗可能静默 no-op 或误操作到变化后的岗位状态”，风险略低，但同样属于 UI 候选集与执行层守门不一致。

### 14. `AppointFlow` 的任命确认没有重新校验“岗位仍可任命 / 候选仍合格”，旧弹窗甚至可能覆盖后来新上的在任者

- 现象：
  - 玩家打开任命面板后，如果某岗位在此期间已被别人占上、或目标角色已失去任命资格，点击任命仍会继续执行。
- 原因：
  - [AppointFlow.tsx](/D:/桌面/Github上传/Wantangfengyun/wantang/src/ui/components/AppointFlow.tsx#L159) 直接调用 `executeAppoint(post.id, targetId, player.id)`。
  - [appointAction.ts](/D:/桌面/Github上传/Wantangfengyun/wantang/src/engine/interaction/appointAction.ts#L121) 进入执行后只检查“候选人仍存活”，不重跑：
    - 该岗位是否仍在玩家可任命范围
    - 候选人当前是否仍满足岗位资格
    - 该岗位是否在弹窗停留期间已被别人占上
  - 如果岗位已被别人占上，[appointAction.ts](/D:/桌面/Github上传/Wantangfengyun/wantang/src/engine/interaction/appointAction.ts#L145) 还会继续按“覆盖任命”逻辑处理旧 holder。
- 影响：
  - 这是这一批里最危险的一条之一：旧任命弹窗可能直接把后来新上的在任者再顶掉。

## 次一级风险

### 1. `CampaignPopup` 在行营状态变化时，子模式仍可能留下旧入口

- 已修主问题：
  - 集结中不可下行军令
  - 行军中不可继续增援
- 仍建议手测：
  - 玩家若已经进入 `addArmy` 子界面，行营在弹窗打开期间切成 `marching`，子界面是否也会同步禁用或退出。
- 原因：
  - 主入口已经补上了 UI 与执行层的一致性，但子模式切换时序仍值得确认。

### 2. `populationSystem / socialSystem` 当前未发现与 04-10 同等级的新裂缝，但死亡后的连带状态仍应继续深挖

- 目前结论：
  - [populationSystem.ts](/D:/桌面/Github上传/Wantangfengyun/wantang/src/engine/systems/populationSystem.ts) 本轮没看到硬问题。
