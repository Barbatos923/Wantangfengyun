# 索引补全与压测方案

**背景**：Gemini建议暂停Phase 3做Engine层全面重构，Claude评估后认为不必要。采取渐进路线：在现有Zustand Store里补齐缺失索引 + 压测验证，然后继续Phase 3。

---

## 第一步：补齐缺失索引（在现有Store里加）

### 1.1 CharacterStore 加 vassalIndex

**问题**：`getVassals(charId)` 目前遍历全部角色找 `overlordId === charId`，O(n)。

**方案**：在 CharacterStore 的 state 里加：
```
vassalIndex: Map<string, Set<string>>   // overlordId → Set<vassalId>
```

**维护时机**：
- 角色创建时，如果有overlordId，加入索引
- `setOverlord(charId, overlordId)` 时，从旧领主Set移除，加入新领主Set
- 角色死亡时，从领主Set移除

**改造**：`postQueries.ts` 的 `getVassals()` 改为从索引查表，O(1)。

### 1.2 TerritoryStore 加 controllerIndex

**问题**：`getTerritoriesByController(charId)` 遍历全部领地，每个领地还要查grantsControl岗位，O(n×k)。

**方案**：在 TerritoryStore 的 state 里加：
```
controllerIndex: Map<string, Set<string>>   // controllerId → Set<territoryId>
```

**维护时机**：
- `updatePost()` 里，如果被更新的Post是 grantsControl 类型的，同步更新 controllerIndex
- 具体：旧holderId的Set移除该territoryId，新holderId的Set加入

**改造**：`postQueries.ts` 的 `getControlledZhou()` 改为从索引查表。

### 1.3 CharacterStore 加 aliveSet

**问题**：`getAliveCharacters()` 每次遍历全部角色过滤 `alive === true`。

**方案**：
```
aliveSet: Set<string>   // 存活角色ID集合
```

**维护时机**：
- 角色创建时加入
- `killCharacter()` 时移除

## 第二步：压测脚本

### 目标
生成5000角色 + 300领地 + 1300岗位，跑12轮月结，验证性能。

### 验收标准
- 12轮月结总耗时 < 500ms（现代浏览器）
- 单轮月结 < 50ms
- 如果超标，再考虑可变状态等进一步优化

### 脚本内容
1. 批量生成角色（随机特质、资源、效忠关系）
2. 生成300个州，每州4个岗位
3. 随机分配岗位持有关系
4. 运行 `runMonthlySettlement()` 12次
5. 用 `performance.now()` 记录每轮耗时

### 放置位置
`src/__benchmarks__/settlement-bench.ts`（不影响正式代码）

---

## 第三步：确认后继续 Phase 3

压测通过后，直接按 Phase3军事系统方案 继续开发。

如果压测不通过（>500ms），再针对性优化：
- 优先看是哪个步骤慢（经济计算？特质处理？领地漂移？）
- 针对热点做局部优化，而非全面重构Engine层

---

## 不做的事情（及原因）

| Gemini建议 | 不做的原因 |
|---|---|
| 创建独立GameEngine单例 | 双数据源同步成本远超收益，回合制游戏不需要 |
| 可变状态计算层 | 5000实体的不可变更新在JS里不是瓶颈 |
| Modifier脏标记系统 | 月结计算量很小，过早优化 |
| UI Selector精细化 | 回合制低频更新，等出现卡顿再做 |
