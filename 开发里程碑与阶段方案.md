# 《晚唐风云》开发里程碑与阶段方案

> **文档目的**：定义从系统设计到可玩版本的完整开发路线图，明确每个阶段的目标、交付物、验收标准与依赖关系。
> **核心原则**：每个阶段结束时都必须产出**可运行、可验证**的成果，而非纯文档。

---

## 总览

```
Phase 0  技术底座与项目骨架          ← 所有后续阶段的前提
   ↓
Phase 1  角色系统 + 领地系统          ← 数值地基
   ↓
Phase 2  官职系统 + 经济循环          ← 权力与资源骨架
   ↓
Phase 3  军事系统                     ← 核心冲突玩法
   ↓
Phase 4  继承系统 + 王朝周期律        ← 跨世代循环
   ↓
Phase 5  AI 史书管线                  ← 核心卖点落地
   ↓
Phase 6  谋略 + 活动 + 派系 + 事件    ← 丰富度与可玩性
   ↓
Phase 7  内容填充与数据制作           ← 历史剧本
   ↓
Phase 8  整合测试与打磨               ← 发布准备
```

---

## Phase 0：技术底座与项目骨架

### 目标
搭建项目工程、核心架构与基础 UI 框架，使后续所有系统开发在统一基础上进行。

### 系统设计工作
- 制定技术规范文档（代码规范、目录结构、状态管理方案）
- 确定三层架构的接口规范（UI层 / 引擎层 / 数据层的边界与通信方式）

### 开发交付物
| 交付物 | 说明 |
|:---|:---|
| 项目工程 | React + TypeScript + Vite + TailwindCSS，目录按三层架构划分 |
| TurnManager 核心 | 回合制状态管理器，支持月推进、事件队列、订阅/通知机制 |
| 基础 UI 框架 | 主界面布局（顶部资源栏、右侧图标栏、地图区占位、底部信息栏），支持面板打开/关闭 |
| SVG 地图原型 | 可拖拽/缩放的地图组件，州级行政区可点击高亮，暂用简化边界 |
| 数据层骨架 | IndexedDB 存档/读档接口，静态注册表加载机制 |
| 时间控制 | 日期显示、速度控制（暂停/1x/2x/3x）、月结算触发 |

### 里程碑验收
> **可以在浏览器中看到主界面布局，点击地图上的州能高亮显示，点击"下一月"能推进游戏日期，存档/读档功能可用。**

## Phase 1：角色系统 + 领地系统

### 目标
实现游戏的两个数值地基——角色与领地。完成后，游戏中应出现有属性的人物和有产出的领地。

### 系统设计工作（进入开发前完成）
- **角色系统详设**：五维能力值的生成规则与范围、完整特质列表及效果定义、好感度计算公式（基础好感 + 特质修正 + 事件修正）、压力系统详细规则、健康值衰减公式、NPC 基础决策逻辑（效用函数框架）
- **领地系统详设**：控制度/发展度/民心的初始值与增减规则、州级产出公式（钱/粮/兵 = f(发展度, 控制度, 建筑, 领主管理能力)）、建筑类型列表与槽位规则、道级/国级的聚合逻辑

### 开发交付物
| 交付物 | 说明 |
|:---|:---|
| Character 数据模型 | 五维能力、特质、好感度矩阵、健康值、压力值、家族关系树 |
| 角色生成器 | 支持按规则随机生成角色（含能力值、特质分配、家族遗传） |
| Territory 数据模型 | 州（含建筑槽、三属性）、道（含治州指定）、国 |
| 月结算：领地产出 | 每月自动计算各州的钱/粮/兵产出，汇总到角色资源池 |
| 月结算：角色状态 | 每月处理健康值变化、压力值变化、年龄增长 |
| 人物信息卡 UI | 点击地图 → 弹出左侧人物卡（能力、特质、资源、关系） |
| 领地信息面板 UI | 点击地图 → 可查看州的三属性、建筑、产出明细 |

### 里程碑验收
> **地图上有数十个州，每个州有实控人（NPC角色）。点击州能看到领主信息和领地属性。点击"下一月"后，资源栏数值会根据领地产出变化。角色会随时间衰老，健康值下降。**

## Phase 2：官职系统 + 经济循环

### 目标
实现职/品双轨制和完整的经济循环，使"当官"和"资源管理"成为实际可操作的玩法。

### 系统设计工作
- **官职系统详设**：贤能值的积累公式与来源清单、品位晋升的贤能阈值表（从九品到一品）、中央/地方全部职位的属性定义（所属机构、品位要求、提供的资源/权力）、法理上下级关系图、任命/罢免/自立的流程与条件、上缴机制的具体比例
- **经济循环详设**：典型藩镇的收支平衡示例、俸禄/军费/建设的具体消耗值、破产机制（资源为负时的惩罚链）

### 开发交付物
| 交付物 | 说明 |
|:---|:---|
| OfficialPosition 数据模型 | 品位表（29阶散官）、职位注册表（中央+地方全部职位） |
| 贤能系统 | 贤能积累、品位自动晋升 |
| 任命系统 | 玩家可任命/罢免下属职位，AI 皇帝可任命中央及地方职位 |
| 上缴与俸禄 | 下属按集权等级上缴钱粮，上级按职位发放俸禄 |
| 经济结算 | 月结算中加入完整的收入/支出计算，资源栏显示每月净变化明细 |
| 官职面板 UI | 查看当前官阶、职位、贤能进度；查看朝廷百官列表 |
| 领地体制面板 UI | 设置/调整集权等级、继承体制、边境战争体制等 |

### 里程碑验收
> **玩家以节度使身份开局，拥有官阶和职位。每月有明确的收入（领地产出 - 下属截留）和支出（俸禄 + 军费）。可以任命/罢免幕僚和下属刺史。贤能积累后可以看到品位晋升。**

## Phase 3：军事系统

### 目标
实现战争的完整流程——从宣战理由到集结、行军、战斗、围城、议和。

### 系统设计工作
- **军队详设**：四种军队（牙兵/官健/乡兵/藩兵）的详细属性表、六兵种的属性值与克制矩阵、牙兵士气系统（赏赐/胜利/待遇的影响公式，兵变触发条件）
- **战斗详设**：战术选项列表与克制关系、地形修正表、将领特质对战斗的加成规则、战争分数计算公式
- **后勤详设**：补给线效率公式、关隘/水道的机制细节

### 开发交付物
| 交付物 | 说明 |
|:---|:---|
| Army 数据模型 | 军队类型、兵种组成、数量、士气、补给状态、将领绑定 |
| War 数据模型 | 战争理由、参战方、战争分数、战争状态机 |
| 宣战流程 | 选择理由 → 检查条件（名望/合法性消耗）→ 正式宣战 |
| 军队集结与行军 | 地图上可视化军队移动，补给线计算 |
| 战斗系统 | 两军相遇 → 关键战役面板 → 战术选择 → 结算 |
| 围城系统 | 攻城进度条，攻城器械加速 |
| 战争结算 | 战争分数达到 100% → 强制投降/议和面板 |
| 牙兵士气系统 | 士气月度变化、赏赐交互、兵变事件触发 |
| 军事面板 UI | 军队列表、战争列表、军队招募/解散界面 |

### 里程碑验收
> **玩家可以向相邻藩镇宣战，集结军队在地图上移动，两军相遇时进入战斗，可围城攻克州城。牙兵需要定期赏赐，赏赐不足会触发兵变。战争胜利后获得领地。**

## Phase 4：继承系统 + 王朝周期律

### 目标
实现跨世代游玩和宏观时代循环，使游戏从"一个人的一生"扩展为"一个家族的数百年"。

### 系统设计工作
- **继承系统详设**：继承法类型（嫡长子继承、指定继承、牙兵推举等）及各自规则、领地分配规则（是否分家）、继承危机的触发条件与流程、好感度继承公式、"留后"机制的详细流程
- **王朝周期律详设**：治世/危世/乱世的全局 buff 具体数值、诱因积分的完整清单及各自权重、时代转换的阈值、"天下"头衔的创建/销毁规则

### 开发交付物
| 交付物 | 说明 |
|:---|:---|
| 继承法系统 | 多种继承法实现，领地体制面板中可切换 |
| 角色死亡与继承流程 | 角色死亡 → 确定继承人 → 领地/职位/资源转移 → 玩家切换视角 |
| 继承危机事件 | 无合法继承人 / 多个竞争者 → 内战/牙兵拥立事件链 |
| 王朝周期律状态机 | 全局 Era 状态，诱因积分月度累积，达阈值触发时代转换 |
| 时代全局效果 | 治世/危世/乱世的 buff/debuff 应用到所有相关系统 |
| 时代转换事件 | 危世→乱世时的"天朝崩解"事件，乱世→治世时的"新朝建立"事件 |

### 里程碑验收
> **玩家角色死亡后，自动切换到继承人视角继续游戏。不同继承法产生不同的继承结果。游戏从危世开始，随着不稳定积分累积，可以观察到时代转换发生，全局规则随之变化。**

## Phase 5：AI 史书管线

### 目标
实现"玩游戏即写史书"的核心卖点——从事件记录到年度筛选到文言文生成的完整管线。

### 系统设计工作
- **事件记录规范**：GameEvent 的完整字段定义、事件类型枚举、重要性评级规则、半文言模板库（每种事件类型对应的描述模板）
- **AI 调用规范**：System Prompt 的最终版本、上下文窗口的组装规则（按重要性筛选+裁剪）、输出格式约束
- **史书体例规范**：编年体/纪传体结构定义、年度史书 vs 一生总结的区别

### 开发交付物
| 交付物 | 说明 |
|:---|:---|
| GameEvent 系统 | 所有已实现系统在关键行为发生时自动写入 GameEvent |
| 年度筛选器 | 每年腊月自动按重要性筛选事件，构建 AI 上下文 |
| ChronicleProvider 抽象层 | 支持 Moonshot / Anthropic / Local 三种后端 |
| AI 文言生成 | 调用 AI 模型，将筛选后的事件转化为编年体文言文 |
| 史书缓存 | 已生成的年度史书存入 IndexedDB，不重复调用 |
| 起居注面板 UI | 可随时查看当年已记录的事件列表（结构化数据） |
| 史书展示面板 UI | 仿古书卷风格，翻页阅读，支持按年份/人物检索 |
| 一生总结 | 角色死亡时，汇总一生事件生成纪传体传记 |

### 里程碑验收
> **玩游戏过程中，起居注面板实时显示已记录的事件。每年年末自动生成一段文言文史书。角色死亡时生成完整的一生传记。史书文风符合"词句锤炼、史官中立、干支纪年"的规范。**

## Phase 6：谋略 + 活动 + 派系 + 事件

### 目标
实现丰富度系统，使游戏从"能跑通"变为"有趣"。

### 系统设计工作
- **谋略系统详设**：个人计谋与政治计谋的完整列表，成功率公式，邀请机制
- **活动系统详设**：个人活动与政治活动的完整列表，效果定义，条件与消耗
- **派系系统详设**：五大派系的详细机制、派系影响力计算、廷议/弹劾/推举的流程
- **事件包设计**：按主题分包（战争、家庭、生活方式、旅行等），每包含若干事件链
- **NPC AI 完善**：基于效用函数的完整 NPC 决策系统（何时发动战争/加入阴谋/叛变/联姻）

### 开发交付物
| 交付物 | 说明 |
|:---|:---|
| 谋略系统 | 计谋发起 → 成功率积累 → 执行/失败，支持多人协谋 |
| 活动系统 | 活动发起 → 持续时间 → 结算效果，含压力释放活动 |
| 派系系统 | 加入/退出派系，派系廷议事件，弹劾/推举流程 |
| 事件系统 | 按条件触发随机事件，支持事件链（多步骤选择） |
| NPC AI 引擎 | NPC 每月自主决策：发起战争/阴谋/外交/活动等 |
| 对应 UI 面板 | 计谋面板、活动面板、派系面板、事件弹窗 |

### 里程碑验收
> **NPC 会主动发起战争、阴谋和外交行动。玩家可以策划暗杀、举办宴会、加入派系。每月有随机事件需要玩家做出选择。游戏世界有了"自己在运转"的感觉。**

## Phase 7：内容填充与数据制作

### 目标
将历史数据注入系统，制作第一个可玩剧本"咸通末年（870年）"。

### 交付物
| 交付物 | 说明 |
|:---|:---|
| 历史地图数据 | 全部州/道的地理边界、法理归属、初始属性 |
| 历史人物数据 | 50-100 位核心历史人物（含能力值、特质、关系网、官职） |
| 历史藩镇数据 | 各藩镇的初始军队配置、领地体制、经济状态 |
| 朝廷初始状态 | 中央百官配置、禁军配置、派系初始势力 |
| 建筑数据 | 全部建筑类型的属性、效果、造价 |
| 事件文本 | 全部事件包的选项文本、结果描述（半文言） |
| 平衡性调参 | 基于测试调整核心公式的参数 |

### 里程碑验收
> **选择"咸通末年"剧本，能看到历史上真实的藩镇格局和人物。以某个历史节度使开局，游玩体验符合历史逻辑（河北藩镇强势、江南富庶、朝廷宦官专权）。**

## Phase 8：整合测试与打磨

### 目标
全面打磨游戏体验，修复系统间的衔接问题，优化性能与 UI。

### 交付物
| 交付物 | 说明 |
|:---|:---|
| 系统联动测试 | 验证所有系统间的数据流转正确（如：战争→领地易手→官职变更→史书记录） |
| 性能优化 | 大量角色/军队同时存在时的帧率保证 |
| UI 打磨 | 动画、音效、视觉一致性、仿古美术风格统一 |
| 新手引导 | 教程事件链，引导玩家理解核心系统 |
| 史书导出 | 支持导出为 PDF / 图片 / 可分享链接 |
| 多剧本支持 | 预留第二剧本入口（如"中和元年/881年·黄巢之乱"） |

### 里程碑验收
> **完整游玩一局（从 870 年到角色死亡且继承人继续游戏至少一代），全程无阻断性 bug，史书完整生成，体验流畅。**

## 阶段间依赖关系总览

```
Phase 0 ──→ Phase 1 ──→ Phase 2 ──→ Phase 3 ──┐
              角色+领地     官职+经济     军事     │
                                                  ↓
                                          Phase 4 继承+周期律
                                                  │
                         Phase 5 ←────────────────┘
                          AI史书（依赖前4阶段的GameEvent产出）
                            │
                            ↓
                         Phase 6 ──→ Phase 7 ──→ Phase 8
                        丰富度系统     内容填充      打磨
```

**关键依赖说明**：
- Phase 1 → 2：官职的产出公式依赖领地系统
- Phase 1 → 3：军队兵源依赖领地属性，将领依赖角色系统
- Phase 2 → 3：战争理由需要消耗名望/合法性
- Phase 1-4 → 5：史书管线需要前四阶段产出 GameEvent
- Phase 5 可以与 Phase 6 并行推进

---

## 每阶段工作流程

每个 Phase 内部遵循统一工作流：

```
系统设计文档 → 评审确认 → 数据模型实现 → 引擎逻辑实现 → UI 实现 → 集成测试 → 里程碑验收
```

**先设计，后编码**：每个阶段开工前，必须先完成该阶段对应的系统设计文档，经确认后再进入开发。这确保开发过程中不会因设计模糊而反复返工。
