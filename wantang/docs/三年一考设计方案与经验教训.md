# 三年一考（考课制度）— 设计方案与经验教训

> 2026-03-29 实施并撤回。本文档记录完整设计、遇到的问题和教训，供后续重新实现时参考。

---

## 一、设计方案

### 1.1 触发方式

统一考课：每三年正月（`date.month === 1 && date.year % 3 === 0`），在 settlement 管线的 socialSystem 之后触发。

### 1.2 适用范围

- 仅 `successionLaw === 'bureaucratic'` 的流官岗位
- 朝廷直辖官员：由吏部/宰相经办（复用 `resolveAppointAuthority`）
- 辟署权范围内的官员：由辟署权持有人自行经办，朝廷不介入

### 1.3 数据扩展

Post 新增字段：

```typescript
reviewBaseline?: {
  population: number;   // 基线人口
  virtue: number;       // 基线贤能
};
reviewBonus?: number;   // 上等加成（+20）
```

每次考课后更新为当前值。初始化时在 `fillVacantSubPosts` 中设置。

### 1.4 考课评分（0–100）

纯函数 `calculateReviewScore(character, territory, baseline, positionTemplate)`：

```
领地人口增长 (权重 0.4):
  无领地岗位（中央京官）→ 65（中性）
  有领地 → popGrowth = (当前人口 - 基线人口) / 基线人口
  领地分 = clamp(popGrowth × 500 + 65, 0, 100)

贤能增长 (权重 0.3):
  virtueGrowth = 当前virtue - 基线virtue
  贤能分 = clamp(virtueGrowth / 3 + 65, 0, 100)

岗位匹配度 (权重 0.3):
  文职取 administration，武职取 military
  匹配分 = clamp(ability × 5, 0, 100)
```

中性基准定为 65（非 50），确保"无功无过"的官员能得到约 63 分（中等留任）。

### 1.5 考课结果

| 分数 | 等级 | 操作 |
|------|------|------|
| ≥ 80 | 上等 | 留任，铨选评分加成 +20 |
| 60–79 | 中等 | 留任，无加成 |
| < 60 | 下等 | 罢免，岗位空缺 |

### 1.6 玩家交互

类似铨选审批（TransferPlanFlow）：
- NPC 经办人评分 → 筛出下等者 → 生成 ReviewPlan
- 玩家是法理主体 → 暂存到 NpcStore.pendingReviewPlan 等待审批
- 否则 → 自动执行罢免（executeDismiss）
- UI：ReviewPlanFlow 审批弹窗，AlertBar 橙色徽章"N项考课待审"
- 罢免时 dismisserId 传法理主体（皇帝/辟署权持有人）

## 二、遇到的问题

### 问题 1：罢免率过高（42%）

**现象**：首次考课 38/91 的流官被罢免。

**原因**：`generateFillerCharacter` 生成的随机角色能力值范围 5-18，很多人 ability ≤ 7，匹配分只有 25-35，拖低总分到 60 以下。

**尝试的修复**：
- 把匹配分公式从 `ability × 5` 改为 `ability × 5 + 15` → 结果几乎没人被罢免了
- 把中性基准从 50 提到 65 → 有改善但不够

**教训**：考课评分公式必须和角色能力分布匹配。如果角色能力平均 10，那 ability=10 应该对应"合格"（60+），而非"勉强及格"。

### 问题 2：人才池枯竭导致连锁循环

**现象**：罢免后大量空缺 → 铨选升调填坑 → 产生新坑 → 下月又要填 → 把吏部尚书都调走了 → 无限循环。

**原因**：
1. 约 160 个角色几乎全有岗位，闲散人才不足 20 人
2. 闲散人才品级集中在 8-15，高品空缺（尚书 20、御史大夫 25）无人可直接新授
3. NPC 只能升调 → 产生新坑 → 连锁不收敛

**尝试的修复**：
- 增加 16 个闲散人才（含 8 个 18-22 品高品）→ 有改善
- 给辟署权节度使各补 3 个人才 → 有改善
- 但到 4-5 年后（自然死亡消耗人口），问题再次出现

**教训**：考课系统不能独立于人才生成系统存在。没有持续补充新人的机制，考课淘汰只会加速人才枯竭。

### 问题 3：高品人才断层

**现象**：空缺岗位品级 20+，但闲散人才最高只有 15-16 品。

**原因**：`generateFillerCharacter` 的默认品级是 8-15。初始设计时没有考虑到需要高品储备。

**教训**：初始人才池需要覆盖所有品级段，尤其是 18-25 品的"废相/致仕大臣"层。

### 问题 4：findEmperorId bug

**现象**：皇帝扮演时考课不弹窗，切换到吏部尚书才弹窗。

**原因**：皇帝岗位已从 centralPosts 移到 tianxia 领地上，但 `resolveLegalAppointer` 仍从 centralPosts 查 pos-emperor，找不到 → fallback 返回经办人而非皇帝 → `legalAppointerId ≠ playerId` → 判断为 NPC 管辖自动执行。

**修复**：创建 `findEmperorId(territories, centralPosts)` 统一查找函数（此修复已保留，不随考课撤回）。

**教训**：已记录到 memory（feedback_emperor_lookup.md）。重要岗位的查找必须用统一函数，不能直接 `.find()`。

## 三、重新实现的前提条件

在重新实装三年一考之前，需要先完成：

1. **人才自然生成系统**（最关键）
   - 每年/每几个月自动生成新的低品级角色（"进士及第"/"举孝廉"）
   - 加入皇帝或各节度使的人才池
   - 品级分布需要覆盖低中高三档
   - 保证活人数量始终 > 岗位数量 × 1.3

2. **品级晋升通道**
   - 低品闲散通过 virtue 积累自然晋升
   - 或者考课上等者获得品级提升
   - 确保高品空缺最终能被低品人才"升上来"

3. **考课评分校准**
   - 评分公式需要基于实际角色能力分布调参
   - 建议：先跑 100 年模拟，统计能力/virtue 分布，再定及格线
   - 罢免率目标：每次考课 5-15%，不超过 20%

4. **NPC 铨选的空缺容忍**
   - 当人才池不足时，NPC 应该能"留空"而非强行升调
   - 避免连锁不收敛

## 五、总结

> 考课制度本质上是一个"消耗型系统"——它消耗人才但不生产人才。
> 在没有配套的"生产型系统"（人才生成）之前，单独实装注定会导致人才枯竭。
> 这是一个**系统依赖**问题，不是考课本身的设计问题。
