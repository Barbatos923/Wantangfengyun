# 《晚唐风云》1.0 开发回顾与经验

> 本文档记录 v1 项目（wantang/）从启动到决定重构的全过程，聚焦**做对了什么、做错了什么、以及为什么要推倒重来**。

---

## 一、项目概况

- **开发周期**：1 天（2026-03-22）
- **技术栈**：React + TypeScript + Vite + TailwindCSS + Zustand
- **完成阶段**：Phase 0（技术底座）+ Phase 1（角色+领地）+ Phase 2（官职+经济）基本完成
- **代码规模**：约 3000 行引擎层 + 1500 行 UI 层 + 1000 行数据层
- **结局**：因架构缺陷导致每次扩展都引发连锁 bug，决定开新项目重构

## 三、做对了什么

### 1. 概念设计先行
在写任何代码之前，先完成了完整的 GDD（16 章概念设计案）和开发里程碑规划。这让开发过程中对"要做什么"始终清晰。

### 2. 三层架构
UI / Engine / Data 的分层在整个开发过程中没有出问题。UI 组件不含业务逻辑，引擎层可独立测试，数据层静态注册表简洁可靠。

### 3. Zustand + Map 存储
用 Map 而非数组存储角色/领地，查找 O(1)，状态更新简洁。

### 4. 交互系统框架
`Interaction` 注册表 + `canShow/paramType` 的设计非常成功——新增"调整集权"交互只需新建一个文件，UI 自动出现在菜单中。这个模式应该在 v2 直接复用。

### 5. 经济概念设计
钱粮双轨、moneyRatio/grainRatio、集权×领地类型上缴率、回拨机制——这些概念设计经过讨论后非常合理，v2 应原样保留。

### 6. Opus 设计 + Sonnet 编码的分工
架构决策和复杂逻辑用 Opus 设计，纯代码实施派 Sonnet agent 并行执行，效率很高。但需要注意 Sonnet 有 Map 当对象用的 bug 倾向。

## 四、做错了什么（核心教训）

### 教训 1：职位模型——模板 vs 岗位

**错误决策**：用一个全局唯一的 `PositionDef`（如 `pos-jiedushi`）作为"节度使"的定义，任命时靠 `PositionHolding.territoryId` 做运行时绑定。

**导致的问题链**：
1. "关内道节度使"和"河东道节度使"不是独立实体，冲突检测需要同时匹配 positionId + territoryId
2. 朱温持有无 territoryId 的节度使 → canAppoint 冲突检测全局误判 → "已有人在任"
3. 任命者自己持有岗位（罢免后自动获得）→ 被自己的冲突检测拦住
4. 每种职位层级（州/道/国/中央）在任命/罢免时都要写特殊分支，分支越加越多

**正确做法**：引入 **Post（岗位）** 概念。岗位 = 职位模板 + 具体领地，预创建在领地上。`holderId` 即真相。任命 = 设 `holderId`，罢免 = 清 `holderId`。零特殊分支。

### 教训 2：领地控制权双源冗余

**错误决策**：同时维护 `character.controlledTerritoryIds`（角色知道自己控制哪些领地）和 `territory.actualControllerId`（领地知道谁控制自己）。

**导致的问题链**：
1. 任命刺史时要更新两处（角色的数组 + 领地的字段），漏一处就不一致
2. `controlledTerritoryIds` 只含州级，不含道/国级，概念不统一
3. 遍历"玩家可见的道级领地"时，道不在 controlledTerritoryIds 中，找不到

**正确做法**：只保留领地上的岗位 `holderId` 作为单一真相源。"谁控制长安" = 长安主岗位的 holderId。"玩家直辖哪些州" = 遍历所有州，找主岗位 holderId === playerId。

### 教训 3：任命流程的两步设计

**错误决策**：任命流程分两步——先选职位模板，再选领地。

**导致的问题链**：
1. 选职位时还没有 territoryId，冲突检测的逻辑与选领地后不同
2. 不同职位的领地选择来源不同（州从 controlledTerritoryIds，道从 positions，国从 controlledTerritoryIds），每种要单独写
3. `getTerritoryRequirement` 函数从 2 个 case 膨胀到 6 个

**正确做法**：直接展示所有**空缺岗位**（holderId === null 且玩家有权任命的）。每个岗位天然就是"职位+领地"的组合，一步完成。

### 教训 4：渐进修补 vs 重新设计

当第一个 bug 出现时（canAppoint 误判），选择了加条件修补。后续每个 bug 都是前一个修补的副作用。修补链：排除 appointee → 加 territoryId 检查 → 区分 scope → 排除 appointer → 仍有问题。

**正确做法**：第一个结构性 bug 出现时就应该回到数据模型层面重新审视。如果模型正确，bug 不应该需要超过 3 行代码来修复。

## 六、v2 架构方向

详见 `官职与经济系统重构草案v2.md`，核心变更：

1. **Post（岗位）模型**替代职位模板 + 运行时绑定
2. **领地控制权单源真相**：主岗位 holderId 推导，移除 controlledTerritoryIds 和 actualControllerId
3. **任命/罢免统一流程**：所有层级走同一函数，grantsControl 标记统一处理
4. **任命 UI 一步完成**：直接展示空缺岗位列表

## 七、工作流经验

1. **Opus 做设计 + Sonnet 做编码**效率很高，但 Sonnet 生成的代码需要特别 review Map 访问方式
2. **编译验证要勤**：每个 batch 改完立即 `npm run build`，不要积累
3. **不要在旧架构上打补丁超过 3 次**：如果同一个区域连续修 3 个 bug，说明模型有问题，应该停下来重新设计
4. **概念设计文档是最值钱的资产**：代码可以重写，但设计认知需要时间积累
