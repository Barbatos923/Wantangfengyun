# 官职与经济系统重构草案 v2

> 本文档是在 v1 经济系统重构方案基础上，针对实现过程中暴露的架构问题进行的全面重新设计。
> 经济系统的**概念设计**（产出公式、集权、上缴率、回拨等）保持不变，本文聚焦于**代码侧数据模型**。

---

## 〇、当前架构的核心问题

### 问题 1: 职位是模板，不是岗位

当前 `pos-jiedushi` 是一个全局唯一的职位模板。但"关内道节度使"和"河东道节度使"是两个不同的**岗位**。任命时靠 `PositionHolding.territoryId` 做运行时绑定，导致：
- 冲突检测逻辑复杂（要同时考虑 positionId + territoryId 组合）
- 没有 territoryId 的持有者（如朱温）引发全局冲突假阳性
- 每种职位在任命/罢免时都要写特殊分支

### 问题 2: 领地控制权双源冗余

`character.controlledTerritoryIds` 和 `territory.actualControllerId` 互为镜像，任何操作都要同步更新两处，漏一处就产生不一致。且 `controlledTerritoryIds` 只含州级，不含道/国级，概念不统一。

### 问题 3: 任命/罢免特殊分支膨胀

刺史、防御使走"州级领地转移"路径；节度使、观察使走"道级领地转移"路径；行台尚书令走"国级领地转移"路径；幕府职位走"关联道"路径；中央职位走"普通"路径。每新增一类职位都要加新分支。

## 一、核心概念重定义

### 1.1 职位模板 (PositionTemplate)

**不变**。和现在的 PositionDef 基本一致，是职位的"种类定义"。

```typescript
interface PositionTemplate {
  id: string;                  // 'tpl-jiedushi', 'tpl-cishi', 'tpl-zaixiang'...
  name: string;                // '节度使', '刺史', '宰相'...
  scope: 'central' | 'local';
  tier?: TerritoryTier;        // local 职位绑定的领地层级: 'zhou' | 'dao' | 'guo'
  territoryType?: TerritoryType; // 'civil' | 'military'
  institution: Institution;
  minRank: RankLevel;
  salary: { money: number; grain: number };
  description: string;
  // 任命此职位时是否转移领地控制权
  grantsControl: boolean;      // true = 刺史/防御使/节度使/观察使/王/行台尚书令
}
```

关键变化：
- `canAppoint` 和 `superiorPositionId` **移除**（移到领地岗位系统中）
- 新增 `tier` 表示绑定的领地层级
- 新增 `grantsControl` 统一标记是否涉及领地转移

### 1.2 岗位 (Post)

**新概念**。岗位 = 职位模板 + 具体挂载位置。是"一个萝卜一个坑"中的**坑**。

```typescript
interface Post {
  id: string;                  // 'post-jiedushi-guannei', 'post-cishi-changan'...
  templateId: string;          // 引用 PositionTemplate.id
  territoryId?: string;        // local 岗位绑定的领地 ID
  holderId: string | null;     // 当前在任者 ID，null = 空缺
  appointedBy?: string;        // 谁任命的
  appointedDate?: GameDate;
}
```

关键设计：
- 岗位在领地创建时就生成，不随任命/罢免创建和销毁
- **冲突检测自动消失**：`holderId !== null` 就是有人在任，不需要遍历所有角色
- 任命 = 设 `holderId`；罢免 = 清 `holderId`
- 一个角色可以持有多个岗位（兼职）

### 1.3 领地 (Territory) — 简化

```typescript
interface Territory {
  // 基础
  id: string;
  name: string;
  tier: TerritoryTier;
  territoryType: TerritoryType;

  // 层级
  parentId?: string;
  childIds: string[];

  // 法理归属
  dejureControllerId: string;

  // ★ 移除 actualControllerId
  // ★ 移除 controlledTerritoryIds (从 Character 上)
  // 实际控制人 = 该领地主岗位(post)的 holderId，动态查询

  // 产出相关 (仅 zhou 有意义)
  basePopulation: number;
  moneyRatio: number;
  grainRatio: number;
  control: number;
  development: number;
  populace: number;

  // 建筑
  buildings: BuildingSlot[];
  constructions: Construction[];
  garrison: number;

  // ★ 岗位列表
  posts: Post[];
}
```

关键变化：
- **移除** `actualControllerId` — 由主岗位的 `holderId` 推导
- **移除** Character 上的 `controlledTerritoryIds` — 通过查询岗位反推
- **新增** `posts` 数组 — 该领地的所有岗位

### 1.4 岗位约定

每个领地在初始化时创建固定的岗位：

**州级 (zhou)**:
- 1 个主岗位：刺史（民事州）或 防御使（军事州）
- N 个副岗位：司马、长史、录事参军...

**道级 (dao)**:
- 1 个主岗位：观察使（民事道）或 节度使（军事道）
- N 个副岗位：节度判官、掌书记、都虞候...（军事道）

**国级 (guo)**:
- 1 个主岗位：行台尚书令（民事）或 王（军事）

**中央**（不挂在领地上，单独存储）:
- 宰相、翰林学士、枢密使 等各 1 个岗位

### 1.5 实际控制人查询

```typescript
// 获取领地的实际控制人 = 主岗位的 holderId
function getActualController(territory: Territory): string | null {
  const mainPost = territory.posts.find(p => {
    const tpl = templateMap.get(p.templateId);
    return tpl?.grantsControl === true;
  });
  return mainPost?.holderId ?? null;
}

// 获取角色直辖的所有州 = 遍历所有州，找主岗位 holderId === charId
function getControlledZhou(charId: string, territories: Map<string, Territory>): Territory[] {
  return [...territories.values()].filter(t =>
    t.tier === 'zhou' && getActualController(t) === charId
  );
}
```

不再需要维护冗余的 `controlledTerritoryIds`。

## 二、任命/罢免统一流程

### 2.1 任命

```typescript
function appointToPost(postId: string, appointeeId: string, appointerId: string): void {
  const post = findPost(postId);
  const tpl = templateMap.get(post.templateId);

  // 1. 设置岗位
  post.holderId = appointeeId;
  post.appointedBy = appointerId;
  post.appointedDate = currentDate;

  // 2. 如果是领地主岗位(grantsControl)，自动处理领地转移
  //    — 不需要任何特殊分支，所有层级统一处理

  // 3. 确保效忠关系
  appointee.overlordId = appointerId;
}
```

### 2.2 罢免

```typescript
function dismissFromPost(postId: string, dismisserId: string): void {
  const post = findPost(postId);

  // 1. 清空岗位
  const previousHolder = post.holderId;
  post.holderId = null;
  post.appointedBy = undefined;

  // 2. 如果是领地主岗位(grantsControl)，罢免者自动接管
  if (tpl.grantsControl) {
    post.holderId = dismisserId;
    post.appointedBy = 'system';
  }
}
```

**零特殊分支**。刺史、节度使、王 全走同一条路径。

### 2.3 任命权

任命权改为基于**层级关系**而非 canAppoint 列表：

```typescript
function canAppointTo(appointer: Character, post: Post): boolean {
  const tpl = templateMap.get(post.templateId);

  // 中央职位：只有皇帝能任命
  if (tpl.scope === 'central') return isEmperor(appointer);

  // 地方主岗位(grantsControl)：当前持有者才能转让
  if (tpl.grantsControl) return post.holderId === appointer.id;

  // 地方副岗位：该领地主岗位持有者可任命
  const territory = findTerritoryByPost(post);
  return getActualController(territory) === appointer.id;
}
```

## 三、效忠关系与经济

这部分概念设计**完全不变**，沿用 v1 方案：

- 集权等级：在 Character 上，`centralization?: CentralizationLevel`
- 回拨率：在 Character 上，`redistributionRate?: number`
- 上缴率：由集权等级 × 领地类型 查表
- 结算流程：自底向上收税 → 自顶向下回拨 → 发薪俸 → 破产检查
- 产出公式：`总产出 = basePopulation × K × (dev/100) × (ctrl/100) × (1 + admin×0.02)`

唯一变化：结算时查"谁控制哪些州"不再读 `controlledTerritoryIds`，而是查岗位。

## 四、Character 类型简化

```typescript
interface Character {
  // 身份 (不变)
  id, name, courtesy, gender, birthYear, deathYear, clan, family

  // 能力 (不变)
  abilities: Abilities;
  traitIds: string[];

  // 状态 (不变)
  health, stress, alive

  // 资源 (不变)
  resources: { money, grain, prestige, legitimacy }

  // 关系 (不变)
  relationships: Relationship[];

  // 效忠
  overlordId?: string;
  centralization?: CentralizationLevel;  // 上级对我的集权
  redistributionRate?: number;           // 我对下属的回拨率

  // 官职
  official?: OfficialData;  // rankLevel, virtue, isCivil

  // 标记
  isPlayer: boolean;
  isRuler: boolean;
  title: string;

  // ★ 移除 controlledTerritoryIds — 通过岗位查询推导
  // ★ OfficialData 中移除 positions[] — 通过岗位查询推导
}
```

角色持有的职位不再存储在角色身上，而是通过查询岗位系统获得：

```typescript
function getHeldPosts(charId: string, allTerritories: Map<string, Territory>): Post[] {
  const posts: Post[] = [];
  for (const t of allTerritories.values()) {
    for (const p of t.posts) {
      if (p.holderId === charId) posts.push(p);
    }
  }
  // + 中央岗位
  for (const p of centralPosts) {
    if (p.holderId === charId) posts.push(p);
  }
  return posts;
}
```

## 六、UI 交互流程简化

### 任命流程

```
1. 点击目标角色 → ⚡ → "任命"
2. 列出当前玩家可任命的所有空缺岗位（遍历玩家控制的领地的 posts，找 holderId === null）
3. 每个岗位显示: "{领地名}{职位名}" (如 "关内道节度使", "长安刺史")
4. 冲突检测: holderId !== null → 自动不显示（已有人在任的岗位不出现）
5. 选择岗位 → 执行任命
```

**没有"先选职位再选领地"的两步流程**。因为每个岗位本身就是"职位+领地"的组合，直接展示所有空缺岗位即可。

### 罢免流程

```
1. 点击目标角色 → ⚡ → "罢免"
2. 列出目标角色持有的、由玩家任命的岗位
3. 选择岗位 → 确认 → 执行罢免
```

## 八、迁移策略

由于这是数据模型层面的重构，建议**一次性替换**而非渐进迁移：

1. 先定义新的类型系统 (PositionTemplate, Post, Territory, Character)
2. 重写 Store (TerritoryStore 管岗位, CharacterStore 精简)
3. 重写任命/罢免逻辑 (统一流程, 零特殊分支)
4. 重写经济结算 (查岗位获取控制关系)
5. 重写 UI 组件 (AppointFlow 简化为单步, CharacterPanel 查岗位显示)
6. 最后迁移初始数据 (sample.ts)

## 九、收益总结

| 问题 | 当前 | 重构后 |
|---|---|---|
| "已有人在任"误判 | 全局遍历 + 复杂条件 | `post.holderId !== null` |
| 领地控制权不同步 | 双源维护 | 岗位即真相，查询推导 |
| 任命/罢免分支 | 5+ 种特殊路径 | 1 个统一函数 |
| 新增职位类型 | 要改 3-4 个文件的分支 | 加一个 PositionTemplate |
| 任命 UI | 两步（选职位→选领地） | 一步（选空缺岗位） |
| 冲突检测 | 50 行 canAppoint | 1 行 holderId check |
